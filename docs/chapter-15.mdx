# 15장 Junit 들여다보기

> Junit은 자바 프레임워크이다. 이 장에서는 Junit 프레임워크에서 가져온 코드를 평가한다. 모듈에 관한 개선인데 내용을 참고해서 정리해보고자한다.

### 보이스카우트규칙에 따르면 우리는 처음 왔을 때보다 더 깨끗하게 해놓고 떠나야한다.

- 오늘날 사용하는 개발환경에서 변수 이름에 범위를 명시할 필요가 없다. 접두어 f를 모두 제거하자.
- 의도를 명확히 표현하려면 조건문을 캡슐화해야한다. 즉, 조건문을 메서드로 뽑아내 적절한 이름을 붙인다.
- 함수에서 멤버변수와 똑같은 변수를 사용하는 이유는 무엇일까? 서로 다른 의미라면 이름은 명확하게 붙인다.
- 부정문은 긍정문보다 이해하기 어렵다. 첫문장 if를 긍정으로 만들어 조건문을 반전한다.
- 함수 이름에 인수를 고려하면 가독성이 훨씬 좋아진다.
- if 문 안에서는 예상 문자열과 실제 문자열을 진짜로 압축한다.
- 멤버 변수 이름도 정확하게 바꾼다.

### 코드가 상당히 깔끔해졌다.

모듈은 일련의 분석 함수와 일련의 조합 함수로 나뉜다. 전체 함수는 위상적으로 정렬했으므로 각 함수가 사용된 직후에 정의된다.
분석 함수가 먼저 나오고 조합 함수가 그 뒤를 이어서 나온다. 그리고 코드를 리팩토링 하다보면 원래했던 변경을 되돌리는 경우가 흔하다.
리팩토링은 코가 어느 수준에 이를때까지 수많은 시행착오를 반복하는 작업이기 때문이다.

### 결론

우리는 보이스카우트규칙을 지켰다. 모듈은 처음보다 조금 더 깨끗해졌다. 원래 깨끗하지 못했다는말이 아니다.
저자들은 우수한 모듈을 만들었다. 하지만 세상에 개선이 불필요한 모듈은 없다. 코드를 처음보다 조금 더 깨끗하게 만드는 책임은 우리 모두에게 있다.
