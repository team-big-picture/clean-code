# 2장 의미있는 이름

> 눈이 무릎까지 쌓일 때쯤이 되면 모두들 두꺼운 외투를 입고 얼굴을 꽁꽁 싸매고 있어 누가 누군지 구분하기 위해서는 이름을 정성스럽게 불러야 했다. 그 때문에 이름에는 '눈이 온 몇 번째 날을 좋아하는'이라는 수식이 붙었다. `<평평한 지구 협회와 두 번째 부고>`

우리들 대다수는 자신이 짠 클래스 이름과 메서드 이름을 모두 암기하지 못 한다. 좋은 이름을 선택하려면 설명 능력이 뛰어나야 하고 문화적인 배경이 같아야 한다. 이것이 제일 어렵다. 좋은 이름을 선택하는 능력은 기술, 비즈니스, 관리 문제가 아니라 교육 문제다. 우리 분야 사람들이 이름 짓는 방법을 제대로 익히지 못하는 이유가 바로 여기에 있다.

## 의도를 분명히 밝혀라

좋은 이름을 지으려면 시간이 걸리지만 좋은 이름으로 절약하는 시간이 훨씬 더 많다. (자신을 포함해) 코드를 읽는 사람이 좀 더 행복해지기 위해 정확한 의도를 가진 이름을 사용해야 한다.

```java
 int d; // 경과 시간(단위: 날짜)
```

이름 d는 아무 의미도 없다. 측정하려는 값과 단위를 표현하는 이름이 필요하다. 가령,

```java
 int fileAgeInDay;
```

의도가 드러나는 이름을 사용하면 코드 이해와 변경이 쉬워진다.

```java
 public List<int[]> getThem() {
   List<int[]> list1 = new ArrayList<int[]>();
   for( int[] x : theList)
    if (x[0] == 4)
     list1.add(x)
  return lsit1;
 }
```

문제는 코드의 단순성이 아니라 **함축성**이다. 즉, 코드 맥락이 코드 자체에 명시적으로 드러나지 않는다.

다음과 같은 정보들이 위 코드에 드러나지 않는다.

- theList에 무엇이 들었는가?
- theList에서 0번째 값이 왜 중요한가?
- 값 4는 무슨 의미인가?
- 함수가 반환하는 리스트 list1을 어떻게 사용하는가?

위 코드를 지뢰찾기 코드를 만든다 생각하고 맥락에 따라 이름을 붙여 다시 작성해보자.

```java
 public List<Cell> getFlaggedCells() {
   List<Cell> flaggedCells = new ArrayList<Cell>();
   for(Cell cell : gameBoard)
    if (cell.isFlagged())
     flaggedCells.add(cell)
  return flaggedCells;
 }
```

적절한 이름만 사용해도 코드 이해가 훨씬 쉬워진다.

## 그릇된 정보를 피해라

프로그래머는 코드에 그릇된 정보를 남겨서는 안 된다. 여러 계정을 그룹으로 묶을 때 실제 `List`가 아니라면 accountList라 명명하지 않는다. 프로그래머에게 List라는 단어는 특수한 의미다. 계정을 담는 컨테이너가 실제 List가 아니라면 프로그래머에게 그릇된 정보를 제공하는 것이다.

또한, 유사한 개념은 유사한 표기법을 사용한다. 이것도 정보다. 일관성이 떨어지는 표기법은 그릇된 정보다.

> 스터디 코멘트

- 일관성을 위해 공백과 들여쓰기의 일관성을 지키는 것이 중요하다.

## 의미 있게 구분하라

연속적인 숫자를 덧붙인 이름(a1, a2...)는 의도적인 이름과 정반대다. 이런 이름은 그릇된 정보를 제공하는 이름도 아니며, 아무런 정보를 제공하지 못하는 이름이다. 저자의 의도가 드러나지 않는다.

> 스터디 코멘트

- 한 글자 변수 (for문에서 i와 같은)는 아무런 의미가 없어 지양하려고 한다.
- 숫자 관련된 정보는 변수로 관리하는 것이 편하다.
- 약어는 최대한 지양하자 (idx 보다는 index로)


## 검색하기 쉬운 이름으로 사용하라

변수 이름의 길이는 범위 크기에 비례해야 한다. (= 전역적으로 쓰일 수록 긴 이름을 넣자)

단수히 '5'라고 코드에 적기보다는 ```const int WORK_DAYS_PER_WEEK =5```가 추후 검색에 용이하다(의도도 들어있음)

## 한 개념에 한 단어를 사용하라

추상적인 개념 하나에 단어 하나를 선택해 이를 고수한다. 똑같은 메서드를 클래스마다 fetch, retrieve, get으로 제각각 부르면 혼란스럽다.

## 해법의 영역에서 가져온 이름을 사용하라

모든 이름을 문제 영역(domain)에서 가져오는 정책은 현명하지 못하다. 전산 용어, 알고리즘 이름, 패턴 이름을 적극 활용하자
